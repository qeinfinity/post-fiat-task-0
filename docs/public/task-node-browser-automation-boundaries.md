# Task Node Browser-Automation Auth Boundaries

Purpose: define safe boundaries for autonomous agents using a browser to help with Post Fiat Task Node evidence submission, verification responses, and signing flows.

This runbook is based on live Task Node browser-flow inspection using public-data tasks. It intentionally omits cookies, auth headers, browser storage, private wallet material, full wallet addresses, private alpha content, and any non-public account data.

## Core Rule

An agent may operate the visible Task Node UI after the human operator has completed authentication and wallet setup, but the agent must not handle secrets, bypass consent gates, or turn private material into public evidence.

## Safe To Observe

An agent may observe and summarize:

- Task title, category, deadline, status, reward amount, verification type, and public task instructions.
- Visible task workflow states such as proposed, accepted, verification, under review, refused, or rewarded.
- Non-sensitive UI mechanics such as request-task categories, add-evidence-item behavior, confirmation pages, and verifier follow-up prompts.
- Public-data evidence generated by the agent, such as public API query snippets, public exchange observations, public URLs, commit hashes, or reproducible terminal summaries.
- Redacted wallet state where needed, for example `<active_wallet>` or `<task_verifier>`, without storing full addresses.

## Never Capture Or Persist

An agent must not read, print, store, submit, or forward:

- Seed phrases, private keys, wallet mnemonics, validator keys, validator tokens, or password material.
- Browser cookies, local storage, session storage, OAuth codes, auth headers, bearer tokens, or raw session payloads.
- Full wallet addresses when a redacted placeholder is sufficient.
- Private alpha, customer flow, employer information, client information, private account data, MNPI, or non-public venue details.
- Raw screenshots containing private account state unless the task explicitly requires a screenshot and the human operator approves the visible contents.

## Authentication Boundary

Human-only:

- Social login selection and OAuth consent.
- Credential entry, passkey use, MFA, and account recovery.
- Optional provider-scope consent, especially any scope that exposes private repository, organization, email, or social-account metadata.

Agent-allowed:

- Navigate to Task Node.
- Confirm that a sign-in page or provider list is visible.
- Stop and ask the operator to complete login when credentials or OAuth authorization appears.
- Resume after the operator says the dashboard is ready.

## Wallet Boundary

Human-only:

- Wallet creation, import, unlock, funding, recovery, mnemonic handling, seed backup, private-key export, or active-wallet switching.
- Any decision that spends meaningful funds or sends PFT outside the standard Task Node evidence/verification transaction.

Agent-allowed:

- Use the already-authenticated UI to submit task evidence when the transaction is the normal nominal Task Node verifier transaction shown by the app.
- Review a confirmation page for non-sensitive fields: task title, artifact count, encrypted storage statement, nominal verifier payment, and network fee.
- Stop if a wallet unlock, mnemonic, seed phrase, private key, funding, or unexpected transfer prompt appears.

## Evidence Submission Boundary

Before submission, the agent should:

1. Re-read the task card and verification instructions.
2. Match the requested evidence type exactly: URL, written proof, alpha response, code, file, screenshot, or GitHub commit.
3. Remove sensitive material and replace unavoidable identifiers with placeholders.
4. For alpha tasks, verify the response is within the requested word count and is free to share.
5. Add the evidence as an evidence item when the UI requires it. In observed alpha flows, the final submit button stayed disabled until `Add evidence item` was clicked.

The agent may submit when:

- The evidence is public, reproducible, and task-relevant.
- Required attestations are true.
- Sensitive/client-confidential checkboxes are left unchecked for public-data evidence.
- The confirmation page shows encrypted storage/IPFS behavior and the normal nominal verifier transaction.

## Verification And Signing Boundary

Verifier follow-ups can appear immediately after submission. The agent may answer if the request can be satisfied from recorded evidence.

Safe verifier responses include:

- Public API snippets used to produce an alpha observation.
- Redacted terminal summaries.
- Public URLs or commit hashes.
- Short clarifications of method, window, assets, venues, or fields.

The agent should not answer by inventing evidence or exposing private state. If the verifier asks for private material, the agent should refuse or provide a sanitized alternative.

After filling a verifier response, the agent may sign and submit only if the app shows the standard nominal verifier transaction and no wallet unlock or secret prompt appears.

## Alpha Shareability Rules

For alpha responses, the agent may tick a no-MNPI/shareability confirmation only when the submitted content:

- Uses public data, local reproducible scanner output, or explicitly authorized operator-provided observations.
- Contains no MNPI, confidential employer information, client information, customer data, private account data, private trading records, or non-public venue details.
- Does not include source-identifying details, private quotes, exact private positions, or uniquely identifying private situations.
- Clearly states evidence boundaries where relevant.

If the answer relies only on public research rather than direct observation, the agent should say so or ask the operator for real observations. It must not fabricate direct experience.

## Public URL Evidence

When a task asks for a public URL:

- Publish only sanitized content.
- Ensure the URL loads without login, paywall, private repo access, or expiring browser state.
- Prefer a markdown page, public gist, public repo file, or public documentation page.
- Do not publish screenshots or browser artifacts containing wallet/session UI unless explicitly approved.

## Minimal Audit Log

For non-trivial Task Node work, record a local memory note with:

- Task ID or redacted task identifier.
- Task title/category and requested evidence type.
- Public sources or commands used.
- Validation performed.
- Submission status and follow-up status.

Do not log cookies, auth state, full browser storage, full terminal scrollback, wallet secrets, private evidence text, or sensitive payloads.
